1. `Test-VmFeaturesEnabled` ŌĆö Calls `Get-WindowsOptionalFeature -Online` for both 'VirtualMachinePlatform' and 'HypervisorPlatform'. Returns `$true` if both have `.State -eq 'Enabled'`. This is the idempotency check (VMFT-02).

Verifies by running `netsh int ipv4 show excludedportrange protocol=tcp` and checking for 58526. Logs WARN (not ERROR) if not confirmed (BOOT-02). 3. `Register-ResumeTask` ŌĆö Takes `[string]$ScriptPath` parameter. Creates scheduled task 'BarakaDeploy-Resume' with:
   - Action: `powershell.exe -NonInteractive -WindowStyle Hidden -ExecutionPolicy Bypass -File "$ScriptPath" -ResumeAfterReboot`
   - Trigger: `-AtStartup`
   - Settings: ExecutionTimeLimit 30 minutes
   - RunLevel: Highest
   - `-Force` flag to overwrite if exists
   Logs registration (BOOT-01).

4. `Invoke-VmFeatures` ŌĆö Main orchestrator (called at bottom of file, guarded by `$env:BARAKA_TEST_MODE`):
   - If `Test-VmFeaturesEnabled` returns `$true`: log "VM features already enabled -- skipping" and `return`. Still call `Invoke-NetshPortReserve` for idempotent port reservation even when features are already enabled.
   - Otherwise: call `Invoke-NetshPortReserve` (BOOT-02), then `Enable-WindowsOptionalFeature -Online -FeatureName 'VirtualMachinePlatform' -All -NoRestart` and same for 'HypervisorPlatform' (VMFT-01).
   - Check `.RestartNeeded` on both results. If either is `$true` (VMFT-03): call `Register-ResumeTask -ScriptPath $PSCommandPath` (but use the deploy.ps1 path, not the step file path ŌĆö get it from `$script:DeployScriptPath` parameter or compute via `Join-Path $PSScriptRoot '..' 'deploy.ps1'`). Then call `Set-DeployState -Name "ResumeStep" -Value "PostVmFeatures"` (BOOT-03). Then call `Restart-Computer -Force`.
   - If neither needs restart: log "Features enabled, no restart needed" and return.

**CRITICAL anti-pattern avoidance (Pitfall 4 from RESEARCH.md):**
The `Invoke-Step` wrapper in Guard.psm1 sets the "VmFeatures-Done" flag ONLY after the body returns. If `Restart-Computer` fires inside the body, the body never returns and the flag is never set. On post-reboot resume, `Invoke-Step` will re-execute the body. This is SAFE because `Test-VmFeaturesEnabled` at the top of the body returns `$true` (features are now enabled), so the body returns immediately. The flag then gets set. This is the designed behavior ŌĆö do NOT try to work around it by setting the flag manually before reboot.

**Bottom of file:** `if (-not $env:BARAKA_TEST_MODE) { Invoke-VmFeatures }` ŌĆö follows Phase 1 pattern from 01-preflight.ps1.
